CI-CD

How is the project setup to use Github actions?

GitHub Actions makes it easy to automate all the software workflows, with world-class CI/CD. Github actions allow to include CI-CD to any of the Github flows. Github actions are event driven that allows to run series of commands after an event occurs

This project is setup in a way where CI-CD capabilities are built in directly in the Github repository. The project has two .yml files - "dev.yml" and "prod.yml". These files are stored in the directory called ".github/workflows". An event triggers the workflow which contains the job. Jobs controls the order in which actions are run and these actions further automate the testing.

For an example, the file "prod.yml" has script that triggers series of actions when a code is pushed to the Github. There are specifically three actions that take place. One action triggers pytest that checks if all the pages are running. Second creates a docker container and pushes to the docker hub. The third action deploys the app into heroku production app. If all the actions run perfectly, Github actions gives green status checks that confirms that the code is safe to be merged.

How you can review code using a development server?

After a pull request is opened, anyone with read access can review and comment on the changes it proposes. You can also suggest specific changes to lines of code, which the author can apply directly from the pull request. Reviewing is done generally looking at the status checks. "Many teams like to make this even more apparent by hooking up a continuous display to the build system - lights that glow green when the build works, or red if it fails are popular."1

how you can deploy master to a production server:

deploying master to a production server is also known as continuous delivery. You achieve continuous delivery by continuously integrating the software done by the development team, building executables, and running automated tests on those executables to detect problems. Furthermore you push the executables into increasingly production-like environments to ensure the software will work in production. extensive automation of all possible parts of the delivery process, is usually done using a DeploymentPipeline.2

How to Deploy project image to Dockerhub when master is updated?

The "prod.yml" file has scripts that trigger actions. One of the script is written to take in the dockerhub login information and docker repository. It then takes the image, builds it and pushes it to the dockerhub repository. So everytime a code is pushed to Github, Github actions trigger the deployment command to the dockerhub repository and that is how the image of the project is deployed to Dockerhub everytime the master is updated.
